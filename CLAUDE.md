# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Python toolkit for scraping search results from OLX.pt (the Portuguese classifieds site): a CLI that exports to CSV/JSON, an HTML viewer, and a PySide6 desktop GUI with local SQLite persistence for tracking listings over time.

Key fact that shapes the whole design: **OLX.pt embeds its full search-results data as JSON** in a `window.__PRERENDERED_STATE__ = "..."` assignment in the page's initial HTML — no JavaScript execution or headless browser is needed. `extract_prerendered_state()` in `scraper.py` extracts and double-decodes it (the JSON document is itself JSON-string-encoded so it can sit inside the `<script>` tag safely). Listings live at `state["listing"]["listing"]["ads"]`; the entire category tree lives at `state["categories"]["list"]` and is embedded on *every* search page regardless of query/category. If OLX changes their frontend to not embed this state, this assumption would need to be re-validated (fetch a search URL with curl and grep for `__PRERENDERED_STATE__` — if it's missing, a different parsing approach or a browser-based tool would be required instead).

## Commands

```bash
# Install dependencies
pip install -r requirements.txt

# Run the scraper CLI (query is a positional arg; --output extension picks the format)
python -m olx_scraper.cli "iphone 13" --pages 3 -o results.csv
python -m olx_scraper.cli "iphone 13" --pages 3 -o results.json

# CLI flags
#   -p / --pages       number of result pages to fetch (default: 1)
#   -o / --output      output file path, must end in .csv or .json (required)
#   --delay            seconds to sleep between page requests (default: 1.0)
#   --category         OLX category, partial name or path (e.g. "telemoveis")
#   --min-price/--max-price/--condition/--sort   filters mirroring olx.pt's own search filters

# Open scraped results as a browsable, sortable HTML page (optionally re-scrape first with -q)
python -m olx_scraper.view results.json
python -m olx_scraper.view results.json -q "iphone 13" --pages 3

# Launch the desktop GUI (search, filter, browse, and save listings to a local SQLite db)
python -m olx_scraper.gui
```

There is no test suite or linter configured yet.

## Architecture

The package (`olx_scraper/`) is split by responsibility:

- `scraper.py` — networking, parsing, and the shared `Listing` model.
  - `fetch_page(query, page, session, *, category_path=None, extra_params=None)`: GETs `https://www.olx.pt/ads/?q=<query>&page=<n>` (the generic `/ads/` endpoint works for any search term across all categories) or, when `category_path` is given, `https://www.olx.pt/{path}/`. Returns raw HTML.
  - `extract_prerendered_state(html)`: locates and double-`json.loads`s the embedded `__PRERENDERED_STATE__` payload (see "What this is" above for why two decode passes are needed).
  - `parse_listings(html)`: extracts one `Listing` per ad in `state["listing"]["listing"]["ads"]`.
  - `fetch_categories(session)` / `find_category(categories, name)`: fetch OLX's full category tree (id → `{id, name, path, level, parentId, children, displayOrder}`) and resolve a user-typed (partial, case-insensitive) name/path to one entry — raising `ValueError` with a list of candidate paths when the term is ambiguous, since OLX addresses categories by URL path slug rather than id.
  - `fetch_regions(session)` / `fetch_cities(session, region_id)`: resolve OLX's location filters — "regions" in OLX's own data are Portugal's districts (plus island groups; 29 total). Unlike categories, no full location taxonomy is embedded on every page: districts only surface as a `metaData.facets.region` facet (`{id, label, count}`) on a generic search page, and cities only surface as `facets.city` once a `search[region_id]` filter narrows results down to a single district — so `fetch_cities` re-runs a search with that filter applied and reads the resulting facet off it (cascading, exactly like category selection cascades in the GUI). There's no name-based resolver like `find_category` for these — callers (the GUI's `LocationPicker`) already hold the OLX id picked from a dropdown.
  - `numeric_price(price)`: pulls a sortable `float | None` out of a pt-PT formatted display price like `"1.199 €"` (`.` = thousands, `,` = decimals); unparseable prices (e.g. "Sob consulta") yield `None` so callers can push them to the end of a sort.
  - `search(query, max_pages, delay, *, category=None, region_id=None, city_id=None, min_price=None, max_price=None, condition=None, sort=None)`: drives pagination using a shared `requests.Session`, resolves `category` via `find_category` and passes `region_id`/`city_id` straight through as OLX's own district/city ids (see `fetch_regions`/`fetch_cities`), passes the rest through as `search[...]` query params (`SORT_OPTIONS` maps friendly `--sort` names like `"newest"`/`"price-asc"` to OLX's `search[order]` values), dedupes by listing `id`, and stops early once a page produces no new listings (signals end of results). Sleeps `delay` seconds between page fetches to be a polite scraper.
  - `Listing` is a `@dataclass` with `id`, `title`, `price`, `city`, `district`, `date` (just the `YYYY-MM-DD` slice of `createdTime`), `url`, `image_url`.
- `export.py` — pure serialization of `list[Listing]` to a file: `to_csv` (via `csv.DictWriter`, fixed `FIELDNAMES` order) and `to_json` (via `json.dump`, `ensure_ascii=False` so accented characters stay readable in the file).
- `cli.py` — `argparse`-based entry point (`main()`); validates the `--output` suffix, calls `search()`, then dispatches to the matching exporter.
- `view.py` — generates a single self-contained HTML page (written to a temp file and opened via `webbrowser.open`) showing scraped listings as a card grid with a client-side JS sort dropdown (by price or date, asc/desc — `price_attr()` wraps `numeric_price` into the page's `data-price` attribute string). Can optionally re-scrape first with `-q`/`--query` (plus `--pages`/`--category`/`--min-price`/etc., mirroring the CLI) and overwrite the input JSON before rendering.
- `gui.py` — PySide6 desktop app (`MainWindow`, launched via `main()`/`python -m olx_scraper.gui`). Three tabs in a `QTabWidget`:
  - **Home tab**: a list of saved searches — `SavedSearchRow` widgets embedded in a `QListWidget` via `setItemWidget` (mirroring how `ListingCard`s are embedded on the Search tab), each showing a one-line summary from `_describe_saved_search` (query, category, district/city, price range if the search predates that filter being commented out, sort, page count, save date — joined with " · ") plus "Run" and "Delete" buttons (double-clicking a row also runs it). `_run_saved_search` switches to the Search tab, restores the query/pages/category/district+city/sort onto the filter bar (`LocationPicker.select_by_ids` handles the async district→city cascade — see "Filter taxonomies" below), and re-runs the search; `_delete_saved_search` removes the row via `db.delete_saved_search` after a `QMessageBox.question` confirmation. The Search tab's "Save search" button persists the current query and filters via `db.save_search`, then disables itself until `_on_search_criteria_changed` fires — which happens whenever the query, pages, category, location, or sort changes, signalling there's something new worth saving.
  - **Search tab**: a filter bar — a bare query `QLineEdit` (placeholder text only; no "Query:"/"Category:" labels), a pages `QSpinBox`, `CategoryPicker` (three always-visible cascading dropdowns mirroring OLX's category/subcategory/sub-subcategory structure), `LocationPicker` (a district + city pair of cascading dropdowns — see "Filter taxonomies" below), a sort-order combo defaulting to "newest", and an icon-only magnifying-glass search button (`QPushButton("🔍")`, fixed width, tooltip "Search") that shows "⏳" while a search is running. The min/max price `QDoubleSpinBox`es are commented out rather than deleted — not needed for now — with a comment block in `_build_search_bar` listing every site to restore (the widgets, their `_on_search_criteria_changed` connections, `_start_search`/`_save_search` kwargs, `_run_saved_search` restoration); `search()`/`db.save_search` receive `min_price=None, max_price=None` meanwhile, and `_describe_saved_search` still renders a price range for any saved search created before this change. The filter bar feeds a background `ScrapeWorker` (runs blocking `search()` on a `QThread` so the UI stays responsive), rendered as a grid of `ListingCard`s (`QListWidget` in `IconMode`). Images load asynchronously via `QNetworkAccessManager` (no extra threads needed — replies deliver on the GUI thread) and are center-cropped to a fixed size with `cover_pixmap()`, cached by listing id in `pixmap_cache` so re-sorting doesn't redownload. Each card overlays three corner badges on its image — a `QCheckBox` selection badge (top-right), a price tag (bottom-right), and a 🔗 "open this listing" button (bottom-left, `open_link_button`/`_open_url`) — as child widgets positioned with `.move()`/`.raise_()` rather than placed in the layout, so none disturb the card's flow. A single click anywhere on the card toggles its selection — mirroring the checkbox badge, the same way clicking a file's icon selects it in a desktop file manager — while a double-click, or a click on the 🔗 badge, opens the listing's URL via `webbrowser.open` (`ListingCard.mousePressEvent`/`mouseDoubleClickEvent`/`_open_url`). The results toolbar's "Ignore selected" button (`_ignore_selected`) hides every ticked card via `db.ignore_listing` and drops it from `self.listings` and the grid — `_on_search_finished` also filters every freshly scraped batch against `db.fetch_ignored_ids` before display, so an ignored ad never resurfaces in a later search either — while "Save selected to database" persists ticked cards via `db.save_listings`.
  - **Database tab** (`DatabaseView`, retitled "Hidden ads" in its mode — see below): a single sortable `QTableWidget` that doubles as both a saved-listings browser and a hidden-ads browser via a "Show hidden ads"/"Show saved listings" toggle button (`_toggle_mode`/`_apply_mode`, `self.showing_hidden`) — swapping `SAVED_COLUMNS`/`HIDDEN_COLUMNS` and the `db.fetch_all`/`db.fetch_ignored` query in place, including a leading thumbnail column, rather than using a second tab or table. Saved mode shows a thumbnail plus title/price/district/posted-date/first-seen/last-seen ordered by `last_seen DESC`; hidden mode swaps in "Hidden on" (ordered by `ignored_at DESC`, most recently hidden first) in place of the seen-tracking columns. Toggling emits `DatabaseView.mode_changed`, which `MainWindow._on_database_mode_changed` uses to rename the tab itself between "Database" and "Hidden ads" so the label always matches what's shown. Thumbnails reuse the `QNetworkAccessManager`/`cover_pixmap()` async-loading pattern from `ListingCard` — sized smaller (`THUMBNAIL_WIDTH`×`THUMBNAIL_HEIGHT`, applied via `QTableWidgetItem.setIcon`/`QIcon`/`table.setIconSize`), cached in their own `thumbnail_cache` (separate from `MainWindow.pixmap_cache`'s larger card-sized crops), and — like the cards — applied by scanning for the matching listing id (`ID_ROLE` stored on each row's logical column-0 item) rather than a remembered row index, so a reply that lands after a sort/refresh still finds the right row. That same logical-column-0 `ID_ROLE`/`URL_ROLE` data is also what `_selected_ids`/`_open_row` key off, keeping selection and double-click-to-open correct even when the user has dragged columns into a different visual order (`header.setSectionsMovable(True)`). Right-clicking the header opens a checkable `QMenu` (`_show_column_menu`) for showing/hiding individual columns; a `_sized_modes` set ensures `resizeColumnsToContents()` runs once per mode after first populating with real data, then leaves the user's manual resizes alone across refreshes. Multi-row selection (`ExtendedSelection`) drives a live "`N` selected" count (`selection_label`/`_update_selection_label`, via `itemSelectionChanged`) and an `action_button` that — after a `QMessageBox.question` confirmation — either permanently deletes the selected rows from `listings` (`db.delete_listings`, "Remove selected") or restores them from `ignored_ads` so they can resurface in future searches (`db.unignore_listings`, "Unhide selected"), with the button's label and the confirmation text both swapping per mode. Refreshes automatically when the tab is selected, right after a save, and right after an ignore action. Double-clicking a row opens its URL in the browser in either mode.
  - **Filter taxonomies**: the category tree, district list, and (cascading from whichever district is picked) city list each load off the GUI thread asynchronously, via the same `QObject`+`QThread`+`moveToThread` worker pattern: `CategoryLoader` feeds `CategoryPicker`; `RegionLoader` (districts are what OLX's own data calls "regions" — see `fetch_regions`) feeds `LocationPicker`'s district combo; and a per-request `CityLoader`, spun up fresh whenever the chosen district changes and tracked in `MainWindow.city_loaders: list[tuple[QThread, CityLoader]]` — a list rather than a single slot, since rapid district switching could otherwise leave stale requests in flight (`LocationPicker.set_cities`/`city_lookup_failed` discard a result whose `region_id` no longer matches the combo's current selection) — feeds the city combo. `LocationPicker.select_by_ids(region_id, city_id)` restores a saved search's district+city despite the async city fetch, by stashing the target city id in `_pending_city_id` until `set_cities` arrives. A shared `filter_status_label` collapses `_categories_ready`/`_regions_ready` into one indicator — "⏳ Loading filters…", "✓ Filters ready" (its tooltip notes that a saved search's "Run" will restore filters correctly once this shows), or "✗ Filters unavailable" with the error surfaced in the status bar/tooltip — generalizing what was originally a category-only readiness check into one that also covers `LocationPicker`'s identical "ran a saved search before its filter data finished loading" race.
- `db.py` — local persistence via stdlib `sqlite3` (db file: `olx_listings.db` in the working directory, `DB_PATH` in `gui.py`). `connect(path)` opens the connection with `row_factory = sqlite3.Row`, ensures all three tables exist (`listings`, `ignored_ads`, `saved_searches`), and runs both column-backfill migrations (`_add_missing_ignored_columns`, `_add_missing_saved_search_columns`). `save_listings(conn, listings)` upserts by `Listing.id` (OLX's own stable ad id): new ads are inserted with `first_seen == last_seen`; existing ads are compared field-by-field across `TRACKED_FIELDS` (normalizing `None`/`""`) — any difference produces a human-readable diff string and an `UPDATE` that refreshes `last_seen`, while unchanged ads just get `last_seen` refreshed. Returns a `SaveResult(added, updated, unchanged, changes)` summary. `fetch_all(conn)` returns every row ordered by `last_seen DESC` for the GUI's Database tab. `ignore_listing(conn, listing)` / `fetch_ignored_ids(conn)` / `fetch_ignored(conn)` back the "ignore this ad" feature — a separate `ignored_ads` table (a near-`listings` snapshot — id, title, price, city, district, date, url, image_url, `ignored_at`) keyed by listing id independent of whether the ad was ever saved, so an ad can be hidden straight from search results without first being added to `listings` and still show up identifiably in the GUI's hidden-ads view later. `ignore_listing` is an upsert (`ON CONFLICT(id) DO UPDATE ...`) — re-ignoring the same id refreshes its snapshot and bumps `ignored_at` rather than erroring or duplicating. `IGNORED_DETAIL_COLUMNS` + `_add_missing_ignored_columns` migrate older databases (created when `ignored_ads` only had `id`/`ignored_at`) by `ALTER TABLE ... ADD COLUMN`-ing the new fields in as nullable, so pre-existing ignored entries keep working with blank details instead of raising "no such column". `delete_listings(conn, ids)` / `unignore_listings(conn, ids)` back the Database tab's "Remove selected"/"Unhide selected" bulk actions — both are parameterized `DELETE ... WHERE id IN (?, ?, ...)` against `listings`/`ignored_ads` respectively (placeholder count comes from the trusted internal id list, not user input), no-op on an empty list, and commit immediately. `save_search(conn, *, query, pages, category_path, category_label, region_id, region_label, city_id, city_label, min_price, max_price, sort)` / `fetch_saved_searches(conn)` / `delete_saved_search(conn, search_id)` back the Home tab's saved-search list — `saved_searches` rows store both the OLX id (`region_id`/`city_id`, `category_path`) needed to re-run a search and the human-readable label (`region_label`/`city_label`, `category_label`) needed to describe it without re-fetching, ordered by `created_at DESC`. `region_id`/`region_label`/`city_id`/`city_label` were added to `saved_searches` after it first shipped — `SAVED_SEARCH_LOCATION_COLUMNS` + `_add_missing_saved_search_columns` backfill them via `ALTER TABLE ... ADD COLUMN` for pre-existing databases (the same approach as `IGNORED_DETAIL_COLUMNS`/`_add_missing_ignored_columns`), so old saved searches keep working with no district/city filter rather than raising "no such column".

Some listings link out to other OLX-Group sites (e.g. `standvirtual.com` for cars, `imovirtual.com` for real estate) rather than to `olx.pt/d/anuncio/...` — `Listing.url` reflects whatever `url` OLX's own ad data contains, which may be a relative OLX path or an absolute external URL.

## Gotchas

- **Windows console encoding**: printing scraped strings (e.g. `€`, accented Portuguese characters, the `→` arrow used in db change-diffs) directly to a `cmd`/PowerShell terminal on Windows can show `�`/raise `UnicodeEncodeError` due to the console's `cp1252` codepage — this is a display artifact only. The actual output files (and the SQLite db) are correctly UTF-8/Unicode; verify by writing to a UTF-8 file (or letting `sqlite3` hold the value) and reading it back rather than trusting console output.
- Always send a realistic browser `User-Agent` (see `USER_AGENT` in `scraper.py`) — OLX may serve a different/degraded response without one.
- Each running GUI instance holds `olx_listings.db` open (`MainWindow.db_connection`, closed in `closeEvent`); relaunching after editing `gui.py`/`db.py` requires killing the previous Python process first since there's no hot-reload.
